Questions

 

local

  • We should define an API and publish it, for the clients to be able to integrate this application with theirs.

    Then, we should model all our services after that API, instead of having a number of tiny interactions.

    The consolidation of the backend actions in a lean small number of basic actions would hide the actual design of the database schema, allowing us to evolve it as needed.

  • In the topics table we need a text instructions field, to be shown to the user, like "enter from 2 to 4 goals in this axis".

    This is needed bacause else the user wouldn't know why the "commit" button is not enabled (like, if she chose 5 goals).

  • No traceability: some tables loaded by the user's interactions lack a timestamp.
  • No change management: the topics table lacks an end date (and perhaps a start date).
  • The tm_translations table should include the organization id in its clave, in order to cater for the different names each company culture assigns to the application objects like "topics" and "competences". Alternatively, something similar could be achieved through configuration, like ODBC does. Fo example in BS the word for "topic" is "eje".
  • The table tm_countries should have a column holding the default language, for global organizations that have employees in different locations. Same for tm_organization_offices.
  • Fields like name in tm_organization_offices should be translatable.
  • The generalized idea is that the user sets a quantitative goal and then she starts to grow a number attempting to reach her goal. But, in some cases it might happen that she starts with a value and tries to reduce it, for example in cases like errors or risks.

    Will the client ask us for diminishing goals?
